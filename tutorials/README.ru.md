# Туториалы — неочевидные инсайты теории очередей для инженера

[🇬🇧 English version](README.md)

Короткие исполняемые ноутбуки (сами ноутбуки — на английском). Каждый на калькуляторах самой
библиотеки (в паре с симуляцией для проверки) показывает результат, который удивляет большинство
инженеров — из тех, что меняют то, как вы масштабируете системы, планируете работу и рассуждаете о
задержке.

### Ёмкость и задержка

| Ноутбук | Инсайт |
|---|---|
| [utilization_trap.ipynb](utilization_trap.ipynb) | Почему **последние 10% загрузки — самые дорогие** (задержка взрывается как `1/(1-ρ)`), и почему **пулинг серверов** бьёт много мелких очередей при той же загрузке. |
| [variability_matters.ipynb](variability_matters.ipynb) | При одном и том же *среднем* времени обслуживания задержка отличается **в 10+ раз** — ожидание растёт с разбросом `C²`, а не только со средним (Полячек–Хинчин). |
| [littles_law.ipynb](littles_law.ipynb) | `L = λW` верно для **любой** системы, дисциплины и распределения — превратите метрику, которую нельзя измерить, в две, которые можно (задержку из конкурентности). |
| [heavy_traffic.ipynb](heavy_traffic.ipynb) | У порога перегрузки время ожидания становится **экспоненциальным при любом распределении обслуживания** — очередь «забывает» форму; важны только первые два момента. |

### Планирование

| Ноутбук | Инсайт |
|---|---|
| [disciplines_comparison.ipynb](disciplines_comparison.ipynb) | Одна лишь **дисциплина** (FCFS, PS, LCFS, FB, SJF, PSJF, SRPT, по предсказанию) сильно двигает среднее время отклика — и меняет, *кто* платит. |
| [srpt_basics.ipynb](srpt_basics.ipynb) | **Планирование по размеру** (SRPT/SJF) резко снижает отклик; предсказания заменяют точные размеры, с плавной деградацией. |
| [conservation_law.ipynb](conservation_law.ipynb) | **Приоритет — игра с нулевой суммой**: слепой классовый приоритет сохраняет взвешенную по загрузке сумму ожиданий `Σ ρ_i W_i` — он перераспределяет задержку, но не убирает её. |

### Ошибки моделирования и парадоксы

| Ноутбук | Инсайт |
|---|---|
| [pasta.ipynb](pasta.ipynb) | **Только пуассоновский вход видит средние по времени** — при регулярном/пачечном трафике то, что видит заявка, отличается от среднего, смещая замеры и формулы. |
| [inspection_paradox.ipynb](inspection_paradox.ipynb) | Случайный наблюдатель **ждёт дольше половины среднего интервала** — вы чаще попадаете в длинные интервалы (`E[X²]/2E[X]`). Парадокс ожидания автобуса. |

### Балансировка нагрузки и современные системы

| Ноутбук | Инсайт |
|---|---|
| [power_of_two_choices.ipynb](power_of_two_choices.ipynb) | Опросить **два** случайных сервера и выбрать короткий — резко лучше одного и почти как опрос всех `N`: «power of two choices» в основе современных балансировщиков. |
| [multiserver_jobs.ipynb](multiserver_jobs.ipynb) | Заявки, занимающие **несколько серверов сразу** (резервирование GPU/ядер): **нельзя использовать `k·μ`** — упаковка и блокировка съедают 20–30% ёмкости. |
| [time_varying_load.ipynb](time_varying_load.ipynb) | **Пик загрузки не совпадает с пиком спроса** — занятая система отстаёт от нагрузки, поэтому штат «по мгновенной интенсивности» приходит с опозданием. Когда PSA, а когда MOL для нестационарной `Mt/M/c`. |
| [age_of_information.ipynb](age_of_information.ipynb) | **Слать обновления чаще может сделать данные старее** — свежесть (Age of Information) U-образна по частоте, с оптимумом посередине. |
| [map_ph_correlation.ipynb](map_ph_correlation.ipynb) | **Коррелированный (пачечный) вход** взрывает задержку даже при той же средней интенсивности — это ловит матрично-аналитический стек MAP/PH. |

Запустить любой можно в Jupyter после `pip install most-queue` (плюс `jupyter`, `matplotlib`,
`pandas`). Все модели описаны в [иллюстрированном каталоге](../docs/models.ru.md).
