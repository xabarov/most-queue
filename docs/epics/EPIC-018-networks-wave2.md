# EPIC-018: Сети МО, волна 2 — блокировки, fork-join в сети, MAP-вход, иллюстрации и туториалы

- **Статус:** proposed
- **Создан:** 2026-07-13
- **Roadmap:** обзор сетей — [../research/queueing-networks-2026.md](../research/queueing-networks-2026.md)
  (резерв п. 6–8); продолжение [EPIC-017](EPIC-017-networks-exact-methods.md)

## Цель

Довести сетевой модуль до практической полноты: сети с конечными буферами
(производственные линии), fork-join внутри маршрутизируемой сети, непуассоновский
(MAP) внешний поток, добор точности из резерва EPIC-017 — плюс визуальная и
образовательная часть: схемы сетевых моделей в каталоге и туториалы с инсайтами.

## Контекст

EPIC-017 закрыл закрытые сети и точные методы (MVA/Бьюзен, Джексон, QNA, G-networks,
BCMP). Из обзора остались направления с высоким практическим спросом (блокировки,
fork-join в сети — 🔥 в trends-2026 §D) и синергия с MAP/PH-стеком (сети принимают
только пуассоновский внешний поток). Сетевые модели — единственные в каталоге без
схем; туториалов по сетям нет (есть только time_varying_load, power_of_two_choices
по смежным темам).

Первоисточники: Dallery–Frein (Operations Research 1993), Brandwajn–Jow (OR 1988),
Balsamo et al. «Analysis of Queueing Networks with Blocking» (Kluwer 2001);
fork-join в сетях — IEEE TPDS 2013 (doi:10.1109/tpds.2013.70), AAP 2004;
Gelenbe 1996 (мультикласс сигналов); Whitt–You 2019 (RQNA).

## Задачи

### П.1 Тандемы и сети с блокировками (конечные буферы)

- [ ] `TandemBlockingCalc` (`theory/networks/blocking.py`): декомпозиция тандема с
      конечными буферами (Brandwajn–Jow / Dallery–Frein), blocking-after-service;
      метрики: пропускная способность, загрузки, вероятности блокировки узлов.
- [ ] Симуляция: расширить `NetworkSimulator` поддержкой конечных буферов узлов с
      блокировкой (BAS; у `NegativeNetwork` параметр `buffers` уже есть — обобщить
      или переиспользовать механизм).
- [ ] Валидация: тандем 2–3 узла vs точная CTMC (малые буферы) и vs симуляция;
      предельные случаи — буфер ∞ сводится к открытой сети.

### П.2 Fork-join внутри маршрутизируемой сети

- [ ] Узел типа fork/join в `theory/networks/`: заявка разветвляется на K подзадач по
      подузлам, собирается на join; аппроксимация времени отклика по TPDS 2013
      (открытые сети) с опорой на существующий `theory/fork_join/`.
- [ ] Симуляция: fork/join-узлы в сетевом симуляторе (у `Task` уже есть сетевые
      атрибуты; синхронизация на join).
- [ ] Валидация: изолированный fork-join узел сводится к `ForkJoinSim`/расчёту;
      сеть с одним fork-join узлом vs симуляция.

### П.3 MAP-вход сети (непуассоновский внешний поток)

- [ ] `NetworkSimulator`: MAP-источник внешнего потока (kendall_notation="MAP" уже
      поддержан в `QsSim` — пробросить в сетевой источник).
- [ ] Аналитика: QNA с c² внешнего MAP-потока (у `OpenNetworkCalcQNA` уже есть
      `arrival_cv2` — добавить хелпер вычисления c² интервалов MAP из MAP/PH-стека) +
      документированное ограничение (QNA не учитывает автокорреляцию, только c²).
- [ ] Валидация: сеть с MAP-входом — QNA(c² MAP) vs симуляция vs QNA(Poisson);
      показать, где двухмоментного приближения хватает, а где корреляция ломает его.

### П.4 Добор точности (резерв EPIC-017)

- [ ] Многоканальные станции в мультичейн-MVA (`BCMPClosedNetworkCalc`) — маргинальные

## Критерии готовности (DoD эпика)

Общий DoD ([../DOD.md](../DOD.md)). Специфично:

- Блокировки: расчёт vs точная CTMC на малых буферах в пределах нескольких процентов;
  buffer → ∞ сводится к открытой сети.
- Fork-join узел в изоляции совпадает с существующим fork-join расчётом/симулятором.
- MAP-вход: QNA(c² MAP) ближе к симуляции, чем QNA(Poisson), на коррелированном входе;
  ограничения задокументированы.
- Каждая схема — в обоих каталогах (EN+RU), в стиле существующих; туториалы
  выполняются end-to-end и добавлены в индекс.
