# EPIC-012: Bulk-service очереди (batching, LLM inference)

- **Статус:** done (2026-07-11)
- **Создан:** 2026-07-10
- **Roadmap:** обзор трендов — [../research/queueing-trends-2026.md](../research/queueing-trends-2026.md)

## Цель

Добавить очереди с **групповым (пакетным) обслуживанием** — сервер обслуживает до `b` заявок за
раз. Актуально из-за batching в LLM-инференсе (время обработки батча зависит от размера батча и
максимальной длины токенов). Дополняет уже реализованный пакетный **приход** (Mˣ/M/1) пакетным
**обслуживанием**.

## Контекст

Классические bulk-service модели (Chaudhry & Templeton, «A First Course in Bulk Queues»):
- **M/M^[b]/1** и **M/M^[a,b]/1** — general bulk service rule (обслуживать, если ≥ a заявок, до b).
- **M/G^[b]/1** — произвольное обслуживание батча; решается через embedded Markov chain / корни
  характеристического уравнения.
- **Mˣ/M/c** — групповой приход + многоканальное обслуживание.

LLM-мотивация: batch processing time = f(размер батча, макс. длина токенов); multi-bin batching
(группировка по предсказанной длине). Модели M/G/1 с bulk service и порогом стабильности по
KV-cache (см. [research](../research/queueing-trends-2026.md)).

## Задачи

- [x] **M/M^[a,b]/1** (`BulkServiceMM1Calc`): точная CTMC над (пакет-в-обслуживании, число
      ожидающих); среднее пребывание/ожидание по Литтлу, распределение числа в системе. Общий
      порог `a` и максимум `b`. b=1 = M/M/1 точно.
- [x] **Bulk-service симулятор** (`BulkServiceSim`): сервер стартует при ≥a, берёт до b; сидируемый.
- [x] **LLM-сценарий**: интенсивность обслуживания как функция размера батча (callable μ(size));
      воспроизведён trade-off throughput/latency и наличие оптимального b (E[T] 1.58→0.71 при b≈8).
- [x] Валидация: b=1 = M/M/1 (точно); M/M^[1,b]/1 и a=b — сверка с симуляцией (~1–4%, сим-шум на
      высокой загрузке); LLM-сценарий calc vs sim. Тесты `test_bulk_service.py` (8 шт.).
- [x] Каталог (EN+RU).
- [ ] *(будущее)* M/G^[b]/1 (общее обслуживание батча через embedded chain), туториал.

## Критерии готовности (DoD эпика)

Общий DoD ([../DOD.md](../DOD.md)). Специфично: при b=1 модель сводится к M/M/1 точно; моменты
совпадают с симуляцией; воспроизведён эффект batching (оптимальный размер батча).

## Результаты

- **`BulkServiceMM1Calc`** — точный M/M^[a,b]/1 (CTMC); поддержка size-зависимой интенсивности для
  LLM-батчинга. **`BulkServiceSim`** — парный симулятор.
- Валидация: b=1 = M/M/1 точно; сверка с симуляцией; LLM trade-off. Тесты (8 шт., зелёные).
- Каталог EN+RU.
